Blog Post 10

How your CSS relates to the HTML and JS

I was fascinated to see how they all integrated in the end, especially regarding the index pages for the blogs and the logs.

Because I could use JavaScript to add classes and IDs, as generalized or as specific as I wanted, it wasn’t necessary for me to write everything out in HTML in order to be able to use the micro formatting to add the CSS needed. The HTML was of course more needed in some other places like in the individual posts, as that is more content-based and cannot be streamlined.

I had to really keep track of which classes and which IDs I used where in order to make the CSS cooperate, however, after going back and adjusting some names and adding some containers, I was able to do this relatively easily by using names that are logical to each section I used them in.

The coolest implementation of the three together was the hamburger menu I made. It was hard, and it gave me more headaches than I thought was necessary, but it is there and it works… I hope. It’s really cool though, because it uses all three coding languages in perfect tandem to create a smaller menu, and when it finally worked, I was rather impressed.

The benefits and challenges of responsive design

The benefits of responsive design were made very evident when I was making this website more responsive. It makes it easier for users to see your website on any device, with any screen size. This makes it easier to get your work out to more people, as not everyone has a computer to view your work on.

When doing responsive design from the start, I imagine that implementing it throughout the website is quite a bit more streamlined, and might be less confusing to do.

This was part of the challenges I faced. It was quite hard to backtrack in order to add responsiveness. And keeping up was a whole other ball game.

The biggest challenge, however, was getting my website to be able to work on all devices. It still doesn’t. When looking at the list of devices you can test your website on in the web inspect element, I definitely do not hit all of them, which is frustrating. The knowledge that the website just does not cooperate on certain devices makes it even more frustrating.

This is something I am definitely coming back to, however, for the last website.

When thinking about how our code matters in decolonialism, can you identify any things in your own coding practice that you should be aware of?

A quote by Flanagan, Howe and Nissenbaum (2005) comes to mind when I think about this: “It is one thing to subscribe, generally, to these ideas, even to make a pragmatic commitment to them, but putting them into practice in the design of technical systems such as software games is not straightforward.” And I think this is very valid to us as developers.

I would love to have my website be accessible to everyone, and perfect in every way, free from bias and usable by everyone. But right now, that isn’t possible. If I were to only ever work on projects like this on my own, without any input from other people who are different from me, I do not think it will ever reach that point.

We have talked many times in class about how a diverse group of people are needed to work on these things in order to find shortcomings, that others might not find.

I can see this on my own website as it is right now. I am trying to get it to a point where everyone can use it without any problems, but it is not there yet, and it might never be. I have biases, even if I do not necessarily know they are there. I try my best to be aware of them and bypass them, but there will always be something I miss. This is why I think it is important to have more people look at it, to point out the things I cannot see.

I think my website is also at the perfect point for me to do this as well, to give it to different people so they can point out where my shortcomings are, so I can make adjustments and improve.

References:

Flanagan, M., Howe, D. C., & Nissenbaum, H. (2005). Values at Play: Design Tradeoffs in Socially-Oriented Game Design. CHI 2005, April 2–7, 2005, Portland, Oregon, USA. ACM Press. DOI: 10.1145/1054972.1055071.
